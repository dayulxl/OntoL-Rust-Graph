# ADR-007: 统一图-描述逻辑映射层 (Unified Graph-DL Mapping Layer)

| 字段 | 值 |
|------|-----|
| **编号** | ADR-007 |
| **日期** | 2026-07-10 |
| **状态** | 生效中 |
| **决策者** | 梁晓亮 |
| **影响范围** | 全量 crate (ontology-storage, ontology-reasoner, ontology-server) |

---

## 背景

OWL2 ↔ 属性图的映射定义分散在 29 个文件中，以 ~150 处硬编码字符串的形式存在。
关系类型（`"INSTANCE_OF"`、`"HAS_PROPERTY"`、`"subClassOf"`）、
节点标签（`"Class"`、`"Individual"`、`"Entity"`、`"Type"`）、
属性键（`"iri"`、`"label"`、`"code"`）重复出现，没有统一的管理。

这导致以下问题：

1. **SHACL 语义验证受阻** — SHACL 引擎不知道 `"iri"` 是 `owl:NamedIndividual` 的标识属性，
   也不知道 `"INSTANCE_OF"` 编码了 `rdf:type`
2. **LLM 工具生成脆弱** — Schema 生成时必须手工维护 OWL 术语与图词汇表的对应关系
3. **运行时 Schema 内省不可行** — 无代码能回答「图的这个关系类型对应哪个 OWL 公理？」
4. **修改成本高** — 改一个术语名需要搜索整个代码库并逐一替换

## 决策

创建一个 **单一事实来源 (Single Source of Truth)** 的映射层：

- **位置**: `crates/ontology-storage/src/mapper/unified_mapping.rs`
- **基准**: [W3C OWL2-to-RDF Mapping](https://www.w3.org/TR/owl2-mapping-to-rdf/)
- **架构**: 三层设计 — namespace URI → 词汇表常量 → 双向查找表 → 便捷函数

### 核心设计的三个层次

**Layer 0** — W3C 命名空间 URI 常量：
- `OWL_NS`、`RDF_NS`、`RDFS_NS`、`XSD_NS`

**Layer 1** — SSOT 图词汇表常量 (`const &str`):
- 节点标签: `CLASS_LABEL`、`INDIVIDUAL_LABEL`、`ENTITY_LABEL`、`TYPE_LABEL` 等
- 关系类型: `INSTANCE_OF_REL`、`SUB_CLASS_OF_REL`、`HAS_PROPERTY_REL` 等
- 属性键: `IRI_KEY`、`LABEL_KEY`、`COMMENT_KEY`、`TRANSITIVE_KEY` 等

**Layer 2** — 双向查找表 (`LazyLock<HashMap<_, _>>`):
- `label` ↔ `OwlEntityType` (节点标签 → OWL 实体类型)
- `rel_type` ↔ `OwlAxiomPredicate` (关系类型 → OWL 公理谓词)
- `prop_key` → `PropertyKeyCategory` (属性键 → 标准分类)

**Layer 3** — 便捷函数:
- `owl_entity_for_label()`、`owl_axiom_for_relation()`、`categorize_property_key()` 等
- 预组合切片: `DOMAIN_LABELS`、`OWL_NODE_LABELS`、`ALL_KNOWN_LABELS`

### 双向映射表

| 方向 | 用途 | 使用者 |
|------|------|--------|
| **DL → Graph** | `ClassExpression` 编译为 `GraphPattern` | DWL2 Query Engine, SWRL Engine |
| **Graph → DL** | 图关系类型反查 OWL 公理类型 | SHACL Engine, LLM Schema Gen, Schema 端点 |

### 依赖

无外部依赖。使用 `std::sync::LazyLock` + `std::collections::HashMap`（Rust 1.80+ std，edition 2024）。

## 替代方案

### 方案 A: 继续硬编码（现状）

**拒绝理由**: 维护成本高，无法支持 SHACL 语义验证和运行时 Schema 内省。

### 方案 B: RDF/OWL 文件 + 序列化/反序列化

**拒绝理由**: 引入 XML/RDF 解析依赖、序列化开销、与属性图模型异构。该项目目标是"图原生"，不是"RDF 兼容"。

### 方案 C: TOML/YAML 外部配置文件

**拒绝理由**: 运行时需要文件 I/O + 解析；编译期无法验证完整性；与 Rust 的零成本抽象理念不符。

## 后果

### 正面

- **可维护性**: 修改一个术语只需改一处
- **SHACL 语义验证**: SHACL 引擎可通过 `owl_axiom_for_relation()` 知道某关系是否编码了 `rdf:type`
- **LLM 友好**: 从 SSOT 生成 JSON Schema 描述 OWL 语义，不再手工维护
- **编译期安全**: `const &str` 常量，错误在编译期暴露
- **零运行时开销**: 性能关键路径直接使用 `const` 常量，无 HashMap 查找

### 负面

- 新增 ~600 行代码的单一文件
- 需要一次性的全 crate 迁移（~150 处替换）
- 下游调用方需额外 `use ontology_storage::mapper::unified_mapping;`

## 迁移策略

分四个阶段执行：

| 阶段 | 范围 | 文件数 | 替换数 |
|------|------|--------|--------|
| Phase A | 创建 SSOT + 注册模块 | 2 | — |
| Phase B | 存储层 (model_mapper, entity, relationship) | 3 | ~28 |
| Phase C | 推理机层 (dwl2, swrl, graph, shacl, actions, behavior, reasoner) | 7 | ~40 |
| Phase D | 服务器层 (routes + main.rs) | 10 | ~30 |
